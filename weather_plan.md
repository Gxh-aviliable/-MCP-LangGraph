# 天气查询优化方案

> 问题：高德地图 MCP 天气服务只能返回 3 天内的天气预报，无法满足长行程或多目的地规划需求。

---

## 一、现状分析

### 当前实现

```
backend/agent/nodes/nodes.py → weather_expert 节点
调用 maps_weather 工具 → 仅返回未来 3 天天气
```

### 限制场景

| 场景 | 问题 |
|------|------|
| 单目的地长行程（如7天） | 第4-7天无天气数据 |
| 多目的地行程（跨多天） | 后段行程无天气预报 |
| 提前很久规划（如一个月后） | 整个行程都在预报范围外 |

---

## 二、方案对比

### 方案 A：组合策略（推荐）

**思路**：实时预报 + 历史气候参考 + 用户提示

```
行程天数 ≤ 3天  → 全部使用高德实时预报
行程天数 > 3天  → 前3天实时预报 + 后续历史气候 + 提示语
```

**优点**：
- 不增加新 API 依赖
- 用户仍能获得有价值信息
- 改动较小，复用现有架构

**缺点**：
- 后续天数非实时预报
- 需要准备历史气候数据源

**改动点**：
1. `weather_expert` 节点判断行程天数
2. 超过 3 天部分使用历史气候数据（可内置或调用外部）
3. 输出结构区分 `forecast_type`

---

### 方案 B：接入第三方天气 API

**可选 API**：

| API | 预报天数 | 免费额度 | 国内覆盖 |
|-----|---------|---------|---------|
| 和风天气 | 7-15天 | 1000次/天 | 好 |
| 心知天气 | 7天 | 400次/天 | 好 |
| OpenWeather | 7天 | 无限制 | 一般 |
| WeatherAPI | 14天 | 100万次/月 | 一般 |

**优点**：
- 实时预报覆盖更长天数
- 数据更准确

**缺点**：
- 需要额外注册获取 API Key
- 增加项目依赖
- 可能产生费用（超出免费额度）

**改动点**：
1. 新增天气 API 配置项
2. 创建新的天气工具或 MCP 服务
3. 在节点中选择调用哪个 API

---

### 方案 C：历史气候数据内置

**思路**：内置中国主要城市的月度气候数据

**数据来源**：
- 中国气象局历史统计数据
- 可从公开数据集获取

**示例数据结构**：
```json
{
  "北京": {
    "4月": {
      "avg_temp": 15,
      "avg_high": 22,
      "avg_low": 8,
      "rainy_days": 5,
      "typical_weather": "晴或多云"
    }
  }
}
```

**优点**：
- 无外部依赖
- 全年可用

**缺点**：
- 需要维护数据文件
- 不是实时预报
- 数据可能过时

---

### 方案 D：动态延迟查询

**思路**：行程生成时不查天气，临出发时再查询

**实现**：
1. 行程生成时标注天气"待确认"
2. 用户出发前通过提醒/推送获取实时天气
3. 需要定时任务机制

**优点**：
- 天气数据最准确

**缺点**：
- 架构复杂度高
- 需要持久化 + 定时任务
- 不适合演示/面试场景

---

## 三、推荐方案详情（方案 A）

### 实现架构

```
┌─────────────────────────────────────────────────┐
│              weather_expert 节点                 │
├─────────────────────────────────────────────────┤
│  1. 计算行程天数 = end_date - start_date         │
│  2. 判断是否 > 3 天                              │
│     ├─ 否 → 调用 maps_weather                    │
│     └─ 是 → 调用 maps_weather + 补充历史气候     │
│  3. 合并数据，标记数据来源                        │
│  4. 添加提示信息                                  │
└─────────────────────────────────────────────────┘
```

### 输出结构变更

**当前**：
```json
[
  {
    "date": "2024-04-01",
    "day_weather": "晴",
    "day_temp": 22
  }
]
```

**改进后**：
```json
[
  {
    "date": "2024-04-01",
    "day_weather": "晴",
    "day_temp": 22,
    "forecast_type": "real",
    "source": "amap"
  },
  {
    "date": "2024-04-05",
    "day_weather": "多云",
    "day_temp": 20,
    "forecast_type": "reference",
    "source": "climate_data",
    "note": "历史气候参考，建议出发前查看实时天气"
  }
]
```

### 需要创建的文件

| 文件 | 用途 |
|------|------|
| `backend/data/climate_data.json` | 中国主要城市月度气候数据 |
| `backend/utils/climate_helper.py` | 历史气候数据查询工具 |

### 改动现有文件

| 文件 | 改动内容 |
|------|---------|
| `backend/agent/nodes/nodes.py` | `create_weather_node` 增加天数判断逻辑 |
| `backend/prompts/__init__.py` | 天气相关提示词增加数据来源说明 |
| `backend/agent/state.py` | `weather_data` 结构增加 `forecast_type` 字段 |

## 五、提示词改动

### WEATHER_AGENT_PROMPT 更新

```
你是天气查询专家。

## 数据来源策略
1. 先调用 maps_weather 获取实时预报（最多3天）
2. 如果行程超过3天，补充历史气候参考数据
3. 输出时标记数据来源：
   - forecast_type: "real" (实时预报) 或 "reference" (历史参考)
   - source: "amap" 或 "climate_data"

## 输出格式
[
  {
    "date": "YYYY-MM-DD",
    "day_weather": "天气",
    "day_temp": 温度,
    "night_temp": 温度,
    "forecast_type": "real/reference",
    "source": "amap/climate_data",
    "note": "如为历史参考，提示用户临出发查看实时天气"
  }
]
```

### PLANNER_AGENT_PROMPT 更新

```
## 天气信息处理
- 如果天气数据包含 forecast_type="reference"，在行程建议中注明
- 提示用户：部分天气为历史参考，建议出发前查看实时预报
- 仍可根据历史气候给出穿衣、防晒等建议
```

## 八、决策建议

### 面试演示场景 → 方案 A（组合策略）

- 不依赖外部服务，稳定可靠
- 展示"处理 API 限制"的工程思维
- 改动可控，不影响核心流程

### 生产环境场景 → 方案 B（第三方 API）

- 数据更准确，用户体验更好
- 需考虑费用和稳定性

### 快速 MVP → 方案 C（纯历史数据）

- 最简单，0 外部依赖
- 牺牲实时性
